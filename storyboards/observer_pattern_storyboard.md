# Observer Pattern — Bilingual Tutorial

A short software-design tutorial that introduces the Observer behavioral
pattern (often at the core of observability/event-driven designs) through a
running example: a `WeatherStation` (Subject) that needs to notify various
displays like a `PhoneDisplay` and `WindowDisplay` (Observers) when the
temperature changes. The video walks from the problem (tight coupling and
hardcoded notifications or polling) to a naive workaround and then to the
Observer solution, shows the class diagram, compares the code before and
after, and closes with the benefits and trade-offs.

The same scene sources render to both landscape and portrait, so the video
doubles as long-form material and as Reels / Shorts content. Narration text for
every scene, and the Manim scene sources themselves, are generated by the
configured `ai_cli` from the descriptions below — the storyboard is
intentionally narration-free. Use semantic colors throughout: DANGER for the
naive/problematic code, OK for the improved Observer design, HIGHLIGHT for the
element currently under discussion, PRIMARY for headline statements, and ACCENT
for section labels.

## Scene: 01_pengantar / Pengantar

**file:** scene_01_pengantar.py
**fallback_duration:** 16

### description
Title card. Introduce the Observer pattern by name and frame it as a behavioral
design pattern whose intent is to define a one-to-many dependency between objects
so that when one object changes state, all its dependents are notified and updated
automatically. Hint at everyday examples — UI event listeners, pub/sub systems,
and data observability pipelines. Keep it clean and inviting, a single headline
plus a one-line subtitle.

## Scene: 02_masalah / Masalah

**file:** scene_02_masalah.py
**fallback_duration:** 26

### description
Problem statement built around a `WeatherStation` that reads temperature and
humidity. Several display devices (`PhoneDisplay`, `WindowDisplay`) need to show
the latest data. Show the `WeatherStation` class containing hardcoded references
to `PhoneDisplay` and `WindowDisplay`. Point out that every time we want to add a
new type of display (e.g., a `SmartWatchDisplay`), we have to modify the core
`WeatherStation` code. Show this tight coupling on screen.

## Scene: 03_pendekatan_naif / PendekatanNaif

**file:** scene_03_pendekatan_naif.py
**fallback_duration:** 30

### description
Demonstrate the naive approaches and why they fall short. First, show polling
(using DANGER color), where displays constantly ask the `WeatherStation` "Did
the data change?", wasting CPU cycles. Next, show the hardcoded notification
where `WeatherStation` calls `phoneDisplay.update()` and `windowDisplay.update()`
directly. Call out the issues: high CPU usage from polling, and rigid, tightly
coupled code that violates the Open/Closed Principle because the subject knows
too much about specific concrete display classes.

## Scene: 04_refactor_observer / RefactorObserver

**file:** scene_04_refactor_observer.py
**fallback_duration:** 30

### description
Introduce the Observer pattern as the solution. Build the architecture using OK
color: define a generic `Observer` interface with an `update()` method. Have the
`WeatherStation` maintain a simple list of `Observer` objects. Show how devices
can dynamically subscribe (`addObserver`) or unsubscribe (`removeObserver`). When
data changes, `WeatherStation` simply loops through its list and calls `update()`
on each observer. Show arrows flowing from the single subject out to multiple
registered observers dynamically.

## Scene: 06_perbandingan_kode / PerbandinganKode

**file:** scene_06_perbandingan_kode.py
**fallback_duration:** 30

### description
Side-by-side comparison of the code before and after applying the Observer,
using code_card panels. On one side (DANGER), show the hardcoded method in
`WeatherStation` calling `phone.update()` and `window.update()` directly. On
the other side (OK), show a clean `for` loop iterating over a list of `Observer`
interfaces: `for observer in observers: observer.update()`. Highlight the simple
difference: before, the subject is hard-wired to concrete classes; after, the
subject is completely decoupled from who is listening.

## Scene: 07_kesimpulan / Kesimpulan

**file:** scene_07_kesimpulan.py
**fallback_duration:** 18

### description
Conclusion. Summarize the Observer intent — define a one-to-many relationship
with automatic notifications — and recap the benefits (loose coupling, Open/Closed
Principle, dynamic subscriptions). Be honest about the trade-offs: the order of
notifications is usually not guaranteed, and careless implementations can lead to
memory leaks if observers are not properly unsubscribed (the "lapsed listener"
problem). End by encouraging its use for event-driven and observability scenarios.

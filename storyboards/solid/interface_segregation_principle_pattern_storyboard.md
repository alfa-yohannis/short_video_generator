# Interface Segregation Principle - Java Bilingual Tutorial

A short software-design tutorial that introduces the Interface Segregation
Principle, the "I" in SOLID, through one running example: an office device that
needs print, scan, and fax capabilities. The video walks from the problem (a
single fat interface that forces every device to implement methods it cannot
support) to the segregated solution, shows the class diagram, compares the code
before and after, and closes with the benefits and trade-offs.

The same scene sources render to both landscape and portrait, so the video works
for short videos and presentation slides. Narration text and Manim scene sources
are generated by the configured `ai_cli` from the descriptions below. Keep every
scene clean, visual, and minimal. Use semantic colors: DANGER for the fat
interface and unsupported methods, OK for the segregated design, HIGHLIGHT for
the current line or capability, PRIMARY for titles, and ACCENT for labels. All
code examples should be written in Java. Use the role names `Machine` for the
fat interface, `Printer`, `Scanner`, and `Fax` for the segregated interfaces,
and `SimplePrinter` and `MultiFunctionPrinter` for the clients.

## Scene: 02_masalah / Masalah

**file:** scene_02_masalah.py
**fallback_duration:** 24

### description
Problem statement built around a Java `Machine` interface that bundles every
capability: `print(Document)`, `scan(Document)`, and `fax(Document)`. Introduce a
`SimplePrinter` class that can only print, yet is forced to implement all three
methods. Show the unused `scan` and `fax` methods filled with
`throw new UnsupportedOperationException()` in DANGER. Show the `Machine`
interface box growing wider as more capabilities are bundled in. Use DANGER
arrows from the unsupported methods to `SimplePrinter`. Make the key problem
clear: a fat interface forces a class to implement methods it does not need.

## Scene: 03_pendekatan_naif / PendekatanNaif

**file:** scene_03_pendekatan_naif.py
**fallback_duration:** 26

### description
Demonstrate the naive approach and why it does not scale. Keep the single big
`Machine` interface and just leave empty bodies or throw exceptions for every
capability a class cannot support. Animate adding a new capability,
`staple(Document)`, to the interface, and show that every implementor must be
opened and changed again, including `SimplePrinter`, which gains yet another
throwing method. Call out only three issues: clients depend on methods they never
call, any change to the fat interface ripples to all implementors, and the type
lies about what it can actually do.

## Scene: 04_segregasi_antarmuka / SegregasiAntarmuka

**file:** scene_04_segregasi_antarmuka.py
**fallback_duration:** 30

### description
Introduce the Interface Segregation Principle as the solution. Build the design
step by step in OK color: split the fat `Machine` interface into small role
interfaces, first `Printer` with `print(Document)`, then `Scanner` with
`scan(Document)`, then `Fax` with `fax(Document)`. Show `SimplePrinter implements
Printer` only, with no unsupported methods. Show `MultiFunctionPrinter implements
Printer, Scanner, Fax`. Animate that each client now depends only on the
capabilities it actually uses. HIGHLIGHT the key idea: many client-specific
interfaces beat one general-purpose interface.

## Scene: 06_perbandingan_kode / PerbandinganKode

**file:** scene_06_perbandingan_kode.py
**fallback_duration:** 28

### description
Comparison of Java code before and after applying the principle, using code_card
panels. In the before panel in DANGER, show the fat-interface version:
`class SimplePrinter implements Machine { void print(Document d) { ... } void
scan(Document d) { throw new UnsupportedOperationException(); } void fax(Document
d) { throw new UnsupportedOperationException(); } }`. In the after panel in OK,
show the segregated version: `class SimplePrinter implements Printer { void
print(Document d) { ... } }`. Highlight the simple difference: before, the class
is forced to implement methods it cannot support; after, it implements only the
single interface it needs.

## Scene: 07_kesimpulan / Kesimpulan

**file:** scene_07_kesimpulan.py
**fallback_duration:** 16

### description
Conclusion. Summarize the Interface Segregation intent: no client should be
forced to depend on methods it does not use. Recap the benefits in short phrases:
smaller focused interfaces, no forced or empty methods, lower coupling, and
changes that no longer ripple across unrelated classes. Be honest about the
trade-offs: more interfaces to manage, and a capable class may implement several
interfaces at once. End with the final formula on screen: `Fat Interface ->
Role Interfaces -> Focused Clients`.

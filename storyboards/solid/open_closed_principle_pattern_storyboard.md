# Open-Closed Principle - Java Bilingual Tutorial

A short software-design tutorial that introduces the Open-Closed Principle, one
of the SOLID principles, through one running example: an area calculator that
must compute the area of many different shapes. The video walks from the problem
(a calculator full of `instanceof` checks that must be edited for every new
shape) to an OCP-compliant solution, shows the class diagram, compares the code
before and after, and closes with the benefits and trade-offs.

The same scene sources render to both landscape and portrait, so the video works
for short videos and presentation slides. Narration text and Manim scene sources
are generated by the configured `ai_cli` from the descriptions below. Keep every
scene clean, visual, and minimal. Use semantic colors: DANGER for rigid
conditional logic that must be modified, OK for the OCP-compliant design,
HIGHLIGHT for the current line or object, PRIMARY for titles, and ACCENT for
labels. All code examples should be written in Java. Use the role names `Client`,
`Abstraction`, and `ConcreteType`.

## Scene: 01_pengantar / Pengantar

### description
Title card. The first scene must display the main title exactly as
`Open-Closed Principle Pattern`. Introduce it as a SOLID design principle whose
intent is that software entities should be open for extension but closed for
modification: you should be able to add new behavior without changing existing,
tested code. Keep the screen simple: one large title in PRIMARY, one short

in ACCENT.

## Scene: 02_masalah / Masalah

**file:** scene_02_masalah.py
**fallback_duration:** 24

### description
Problem statement built around a Java `AreaCalculator` class that computes the
area of a shape. The class contains hardcoded type checks:
`if (shape instanceof Rectangle)`, `else if (shape instanceof Circle)`, and
`else if (shape instanceof Triangle)`. Show the `AreaCalculator` box growing
wider as each new shape rule is added. Use DANGER arrows from the
`AreaCalculator` class to each branch. Make the key problem clear: every new
shape forces us to open and edit the same class that already works.

## Scene: 03_pendekatan_naif / PendekatanNaif

**file:** scene_03_pendekatan_naif.py
**fallback_duration:** 26

### description
Demonstrate the naive approach and why it does not scale. Show a method
`double totalArea(List<Object> shapes)` with a long `if/else` chain of
`instanceof` checks, each computing a formula inline in DANGER. Animate adding a
new option, `Hexagon`, and show that the same method must be opened and changed
again, with HIGHLIGHT on the freshly inserted branch. Call out only three issues:
the code violates the Open-Closed Principle, each branch is harder to test in
isolation, and the `AreaCalculator` class now knows too much about every shape's
formula.

## Scene: 04_refactor_ocp / RefactorOCP

**file:** scene_04_refactor_ocp.py
**fallback_duration:** 30

### description
Introduce the OCP-compliant design as the solution. Build it step by step in OK
color: first define a stable `Shape` abstraction (interface) with
`area(): double`, then show three concrete types: `Rectangle`, `Circle`, and
`Triangle`, each carrying its own `area()` implementation. Show the
`AreaCalculator` Client depending only on `Shape` and delegating with the key
line `shape.area()`. Animate adding a new `Hexagon` type that implements `Shape`,
and emphasize that the `AreaCalculator` stays closed and untouched while the
system is extended.

## Scene: 06_perbandingan_kode / PerbandinganKode

**file:** scene_06_perbandingan_kode.py
**fallback_duration:** 28

### description
Side-by-side comparison of Java code before and after applying the Open-Closed
Principle, using code_card panels. In one panel in DANGER, show:
`if (shape instanceof Circle) return Math.PI * c.r * c.r;`. In the other panel in
OK, show:
`class Circle implements Shape { public double area() { return Math.PI * r * r; } }`
together with the client line `total += shape.area();`. Highlight the simple
difference: before, the Client decides every shape's formula with type checks and
must be modified for each new shape; after, each shape carries its own formula
behind the same `Shape` abstraction and the Client never changes.

## Scene: 07_kesimpulan / Kesimpulan

**file:** scene_07_kesimpulan.py
**fallback_duration:** 16

### description
Conclusion. Summarize the Open-Closed Principle intent: open for extension, closed
for modification. Recap the benefits in short phrases: fewer conditionals, isolated
and easier testing, safer changes to working code, and new types without editing
the Client. Be honest about the trade-offs: more small classes and the need to
choose the right abstraction up front. End with the final formula on screen:
`Client -> Abstraction <- ConcreteType`.

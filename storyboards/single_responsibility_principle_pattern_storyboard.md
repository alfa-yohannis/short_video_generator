# Single Responsibility Principle - Java Bilingual Tutorial

A short software-design tutorial that introduces the Single Responsibility
Principle, the first of the SOLID principles, through one running example: an
`Invoice` class that calculates its own total, saves itself to a database, and
prints a formatted report. The video walks from the problem (one class with many
unrelated reasons to change) to the SRP solution that splits those duties into
focused classes, shows the class diagram, compares the code before and after, and
closes with the benefits and trade-offs.

The same scene sources render to both landscape and portrait, so the video works
for short videos and presentation slides. Narration text and Manim scene sources
are generated by the configured `ai_cli` from the descriptions below. Keep every
scene clean, visual, and minimal. Use semantic colors: DANGER for the overloaded
class with mixed duties, OK for the SRP design, HIGHLIGHT for the current line or
object, PRIMARY for titles, and ACCENT for labels. All code examples should be
written in Java. Use the role names `Invoice`, `InvoiceRepository`, and
`InvoicePrinter`.

## Scene: 01_pengantar / Pengantar

**file:** scene_01_pengantar.py
**fallback_duration:** 16

### description
Title card. The first scene must display the main title exactly as
`Single Responsibility Principle Pattern`. Introduce it as the first of the SOLID
design principles whose intent is simple: a class should have only one reason to
change, meaning it should own exactly one responsibility. Keep the screen simple:
one large title, one short subtitle, and three compact icons labeled Calculate,
Save, and Print to hint at the duties that will later be separated.

## Scene: 02_masalah / Masalah

**file:** scene_02_masalah.py
**fallback_duration:** 24

### description
Problem statement built around a Java `Invoice` class that tries to do
everything. Show the class holding three unrelated methods:
`Money calculateTotal()`, `void saveToDatabase(Connection conn)`, and
`void printInvoice()`. Show the `Invoice` box growing taller as each duty is
stacked inside it. Use DANGER arrows pointing from the single class to three
different reasons to change: new tax rules, a new database, and a new report
format. Make the key problem clear: one class carries three independent reasons
to change.

## Scene: 03_pendekatan_naif / PendekatanNaif

**file:** scene_03_pendekatan_naif.py
**fallback_duration:** 26

### description
Demonstrate the naive approach and why it does not scale. Show the `Invoice`
class with a long mix of business logic, SQL strings, and formatting code in one
place. Animate adding yet another duty, `void sendByEmail()`, and show that the
same crowded class must be opened and edited again. Call out only three issues:
the class violates the Single Responsibility Principle, the calculation cannot be
tested without dragging in the database and printer, and a change to the report
format risks breaking persistence or the totals.

## Scene: 04_refactor_srp / RefactorSRP

**file:** scene_04_refactor_srp.py
**fallback_duration:** 30

### description
Introduce SRP as the solution. Build the design step by step in OK color: split
the overloaded class into three focused classes. Keep `Invoice` holding the line
items and `Money calculateTotal()`, the pure business logic. Move persistence
into `InvoiceRepository` with `void save(Invoice invoice)`. Move presentation
into `InvoicePrinter` with `String format(Invoice invoice)`. Animate each
extracted duty sliding out of the crowded class into its own clean box, and label
each new class with the single reason it would ever change.

## Scene: 06_perbandingan_kode / PerbandinganKode

**file:** scene_06_perbandingan_kode.py
**fallback_duration:** 28

### description
Side-by-side comparison of Java code before and after applying SRP, using
code_card panels. In the first panel in DANGER, show one class mixing duties:
`class Invoice { Money calculateTotal() {...} void saveToDatabase() {...} void
printInvoice() {...} }`. In the second panel in OK, show the same work split
across focused classes: `class Invoice { Money calculateTotal() {...} }`,
`class InvoiceRepository { void save(Invoice invoice) {...} }`, and
`class InvoicePrinter { String format(Invoice invoice) {...} }`. Highlight the
simple difference: before, one class changes for many reasons; after, each class
changes for only one.

## Scene: 07_kesimpulan / Kesimpulan

**file:** scene_07_kesimpulan.py
**fallback_duration:** 16

### description
Conclusion. Summarize the SRP intent: a class should have only one reason to
change. Recap the benefits in short phrases: focused classes, easier testing,
lower coupling, and changes that stay contained. Be honest about the trade-offs:
more small classes and the need to wire them together through a coordinator or
caller. End with the final formula on screen:
`One Class -> One Responsibility -> One Reason to Change`.

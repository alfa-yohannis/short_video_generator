# Singleton Pattern — Bilingual Tutorial

A short software-design tutorial that introduces the Singleton creational
pattern through a single running example: an application configuration manager
(`AppConfig`) that must have exactly one shared instance. The video walks from
the problem (scattered, inconsistent instances) to a naive workaround and then
to the Singleton solution, shows the class diagram, compares the code before and
after, and closes with the benefits and trade-offs.

The same scene sources render to both landscape and portrait, so the video
doubles as long-form material and as Reels / Shorts content. Narration text for
every scene, and the Manim scene sources themselves, are generated by the
configured `ai_cli` from the descriptions below — the storyboard is
intentionally narration-free. Use semantic colors throughout: DANGER for the
naive/problematic code, OK for the improved Singleton design, HIGHLIGHT for the
element currently under discussion, PRIMARY for headline statements, and ACCENT
for section labels.

## Scene: 01_pengantar / Pengantar

**fallback_duration:** 16

### description
Title card. Introduce the Singleton pattern by name and frame it as a creational
design pattern whose intent is to guarantee that a class has exactly one instance
while providing a single, global point of access to it. Hint at the everyday
examples — an application configuration, a logger, a connection pool — and set up
the learning goal for the next couple of minutes. Keep it clean and inviting, a
single headline plus a one-line subtitle.

## Scene: 02_masalah / Masalah

**file:** scene_02_masalah.py
**fallback_duration:** 26

### description
Problem statement built around an application configuration manager `AppConfig`
that holds settings such as a theme, a language, and an API endpoint. Several
parts of a program each create their own `AppConfig` with `new`. Show two or
three independent components each holding a separate `AppConfig` object, so that
a change made through one is invisible to the others and memory is wasted on
duplicates. Pose the question on screen: how do we make every part of the program
share one and the same configuration object, with one consistent state?

## Scene: 03_pendekatan_naif / PendekatanNaif

**file:** scene_03_pendekatan_naif.py
**fallback_duration:** 30

### description
Demonstrate the naive approaches and why they fall short. Show creating
`AppConfig` objects with `new` in multiple places (using DANGER color) and the
resulting inconsistent state, then show the common workaround of passing one
`AppConfig` instance through constructors and method parameters everywhere
("parameter drilling"), and the alternative of a mutable global variable that
anyone can reassign. Call out the issues: no real guarantee of a single instance,
threading the object through the whole codebase, hidden coupling, and fragile
global state. Make it clear the language itself does not stop a second `new`.

## Scene: 04_refactor_singleton / RefactorSingleton

**file:** scene_04_refactor_singleton.py
**fallback_duration:** 30

### description
Introduce the Singleton as the solution. Build the `AppConfig` class up using OK
color: a private constructor so no other code can call `new`, a private static
field that holds the one instance, and a public static accessor
`getInstance()` (lazily creating the instance on first call, then returning the
same object every time). Show two callers invoking `AppConfig.getInstance()` and
both arrows pointing to the single shared object. Mention, briefly, that in a
multi-threaded program the lazy creation must be made thread-safe.

### description

## Scene: 06_perbandingan_kode / PerbandinganKode

**file:** scene_06_perbandingan_kode.py
**fallback_duration:** 30

### description
Side-by-side comparison of the code before and after applying the Singleton,
using code_card panels. On one side (DANGER), creation scattered across the
program with `config = new AppConfig()` in several places and the object passed
through parameters. On the other side (OK), a private constructor plus
`config = AppConfig.getInstance()` used uniformly, every call returning the same
object. Highlight the two differences: uncontrolled `new` versus a controlled
static accessor, and many independent instances versus one shared instance.

## Scene: 07_kesimpulan / Kesimpulan

**file:** scene_07_kesimpulan.py
**fallback_duration:** 18

### description
Conclusion. Summarize the Singleton's intent — exactly one instance with a single
global access point — and recap the benefits (consistent shared state, controlled
access, lazy initialization). Be honest about the trade-offs: it introduces
global state, can hide dependencies and make unit testing harder, and needs care
to be thread-safe. End by encouraging viewers to reach for a Singleton only when a
single shared instance is genuinely required.

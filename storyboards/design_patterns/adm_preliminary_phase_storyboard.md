# ADM Preliminary Phase

A short tutorial on the Preliminary Phase of the TOGAF Architecture Development
Method (ADM). TOGAF is a method, not a drawing notation, so every scene is built
from the ADM cycle and a process flow of inputs, activities, and deliverables -
not from ArchiMate boxes and relationships, and not from program code. The running
example is small and familiar: a neighbourhood food-delivery service called
QuickBite that has never done architecture before and now wants to set up the
practice. Keep the TOGAF terms in English (Architecture Vision, Business
Architecture, Gap Analysis, Architecture Definition Document, Stakeholder,
Architecture Principles, Request for Architecture Work) and name the example's own
details in English. Use PRIMARY for titles, ACCENT for phase and deliverable
labels, HIGHLIGHT for the current step, OK for a finished, agreed result, and
DANGER for the gap the phase is meant to remove. Depict the ADM cycle accurately -
the correct phase names in their correct order, Preliminary then A through H with
Requirements Management at the hub - and use clear process/flow diagrams (inputs to
activities to deliverables), never an invented notation. The English version
should all be in English, including the examples.

## Where It Sits in the ADM (~20s)
Show "ADM Preliminary Phase" as a LARGE CENTERED title with a short subtitle
beneath it. Draw the ADM as a cycle of phases - Preliminary, then A (Architecture
Vision), B (Business Architecture), C (Information Systems Architectures), D
(Technology Architecture), E (Opportunities and Solutions), F (Migration Planning),
G (Implementation Governance), and H (Architecture Change Management), with
Requirements Management at the hub - and HIGHLIGHT the Preliminary Phase. Note that
it is the entry point of the whole method: it runs once to set things up, before
the first turn of the cycle into Phase A.

## What the Preliminary Phase Is For (~24s)
State the purpose: the Preliminary Phase prepares the organisation to do
architecture well. It answers the framing questions - Where we do architecture,
What it covers, Why we are doing it, Who will do it, and How we will do it - and
stands up the Architecture Capability itself. Mark the starting situation for
QuickBite in DANGER: no agreed way of working, no defined team, and every decision
made ad hoc. The phase replaces that with an agreed foundation, shown building
toward OK, before any actual architecture is produced.

## Inputs (~22s)
Lay out what flows into the Preliminary Phase as ACCENT-labelled sources. HIGHLIGHT
the key difference from later phases: because nothing in the ADM cycle has run yet,
the inputs do not come from earlier ADM deliverables - they come from the
surrounding enterprise. Show them flowing in: board strategies and business plans,
the business and IT strategy, existing governance and legal frameworks, any major
frameworks already in use, and the TOGAF framework itself as the starting method to
be tailored.

## Key Steps (~30s)
Walk the activities as a process flow, one HIGHLIGHTed at a time, each feeding the
next in OK: scope the parts of the enterprise that the architecture will touch;
confirm the governance and support frameworks; establish the Architecture Team and
its organisational model; identify and agree the Architecture Principles that will
guide every later choice; tailor TOGAF and any other selected frameworks to fit the
organisation; and select and implement the architecture tools and repository.
Emphasise that these steps create a capability, not a finished architecture.

## Deliverables (~24s)
Assemble the outputs the phase produces, building them up in OK as ACCENT labels:
the Organisational Model for Enterprise Architecture, a Tailored Architecture
Framework that includes the agreed Architecture Principles, an initial populated
Architecture Repository, and a restatement of the business principles, goals, and
drivers. HIGHLIGHT the final deliverable - a Request for Architecture Work - which
authorises and triggers the first real turn of the cycle into Phase A.

## How It Connects (~20s)
Place the Preliminary Phase back in the cycle. Its Request for Architecture Work
flows forward into Phase A (Architecture Vision), and its Architecture Principles,
tailored framework, and repository become the shared foundation that every later
phase relies on. Show that the governance set up here is what feeds and checks
Requirements Management at the hub. Stress the direction of flow: the Preliminary
Phase hands off into Phase A and then steps aside, running again only when the
capability itself must be renewed.

## A Worked Example (~24s)
Walk QuickBite through the phase concretely, building each result in OK. Scope: the
delivery operations and the partner restaurants it serves. Team: a small
Architecture Team reporting to the founder as a Stakeholder. Principles: "Customer
ordering comes first," "Reuse before build," and "One source of menu data."
Tailored framework: a trimmed-down TOGAF with a light governance board suited to a
small business. Repository: a shared folder seeded with current system notes.
Finally, a Request for Architecture Work to redesign the ordering experience,
HIGHLIGHTed as it launches Phase A.

## Recap (~16s)
Close by recapping the flow of the phase on screen: Enterprise Inputs -> Scope ->
Governance and Team -> Architecture Principles -> Tailored Framework and Repository
-> Request for Architecture Work. Show the ADM cycle once more with the Preliminary
Phase in OK and an arrow turning into Phase A, reinforcing that this one-time setup
is what makes every later phase possible.

# Strategy Pattern - Java Bilingual Tutorial

A short software-design tutorial that introduces the Strategy behavioral pattern
through one running example: an online checkout that needs to calculate shipping
costs using different algorithms such as standard, express, and same-day
shipping. The video walks from the problem (large conditionals inside the
checkout) to the Strategy solution, shows the class diagram, compares the code
before and after, and closes with the benefits and trade-offs.

The same scene sources render to both landscape and portrait, so the video works
for short videos and presentation slides. Narration text and Manim scene sources
are generated by the configured `ai_cli` from the descriptions below. Keep every
scene clean, visual, and minimal. Use semantic colors: DANGER for rigid
conditional logic, OK for the Strategy design, HIGHLIGHT for the current line or
object, PRIMARY for titles, and ACCENT for labels. All code examples should be
written in Java. Use the role names `Context`, `Strategy`, and
`ConcreteStrategy`.

### description
Title card. The first scene must display the main title exactly as
`Strategy Pattern`. Introduce it as a behavioral design pattern whose intent is
to define a family of algorithms, put each one in its own class, and make those
algorithms interchangeable. Keep the screen simple: one large title, one short
subtitle, and three compact shipping icons labeled Standard, Express, and
Same-Day.

## Scene: 02_masalah / Masalah

**file:** scene_02_masalah.py
**fallback_duration:** 24

### description
Problem statement built around a Java `Checkout` class that calculates shipping
cost. The class contains hardcoded conditional branches:
`if (type.equals("standard"))`, `else if (type.equals("express"))`, and
`else if (type.equals("sameDay"))`. Show the `Checkout` box growing wider as new
shipping rules are added. Use DANGER arrows from the `Checkout` class to each
branch. Make the key problem clear: every new algorithm forces us to edit the
same class.

## Scene: 03_pendekatan_naif / PendekatanNaif

**file:** scene_03_pendekatan_naif.py
**fallback_duration:** 26

### description
Demonstrate the naive approach and why it does not scale. Show a method
`calculateShipping(Order order, String type)` with a long `if/else` or `switch`.
Animate adding a new option, `pickup`, and show that the same method must be
opened and changed again. Call out only three issues: the code violates the
Open/Closed Principle, each branch is harder to test in isolation, and the
`Checkout` class now knows too much about every shipping algorithm.

## Scene: 06_perbandingan_kode / PerbandinganKode

**file:** scene_06_perbandingan_kode.py
**fallback_duration:** 28

### description
Side-by-side comparison of Java code before and after applying Strategy, using
code_card panels. On the left side in DANGER, show:
`if (type.equals("express")) return expressCost(order);`. On the right side in
OK, show:
`class ExpressShipping implements ShippingStrategy { Money cost(Order order) {
return expressCost(order); } }` and the client line
`checkout.setStrategy(new ExpressShipping());`. Highlight the simple difference:
before, the Context chooses every algorithm with conditionals; after, each
algorithm lives behind the same Strategy interface.

## Scene: 07_kesimpulan / Kesimpulan

**file:** scene_07_kesimpulan.py
**fallback_duration:** 16

### description
Conclusion. Summarize the Strategy intent: encapsulate interchangeable
algorithms behind one interface. Recap the benefits in short phrases: fewer
conditionals, easier testing, runtime flexibility, and new algorithms without
editing the Context. Be honest about the trade-offs: more small classes and the
Context or caller must choose the correct strategy. End with the final formula on
screen: `Context -> Strategy -> ConcreteStrategy`.
